# Sourcerer — V1 Test Protocol

## Purpose

This document defines how we measure the three V1 success metrics:
1. **Tier 1 precision:** >70% of Tier 1 candidates are human-validated as strong fits
2. **Dedup accuracy:** <5% duplicate rate across multi-source runs
3. **Cost per run:** <$5 for a typical 50-candidate search (Exa + AI, excluding premium adapters)

---

## 1. Tier 1 Precision Testing

### Benchmark Role

Use `test-fixtures/benchmark-role.json` — "Senior Backend Engineer for a DeFi startup" at Lunar Labs.

This role was chosen because:
- Technical skills are objectively verifiable (Go commits, OSS repos)
- Domain relevance is measurable (DeFi/crypto experience)
- The scoring rubric has clear signal dimensions

### Test Runs

Run the full pipeline 3 times with the benchmark role:
1. **Run A:** Default search config as generated by intake
2. **Run B:** Same role, adjusted weights (technical 40%, domain 20%)
3. **Run C:** Same role, different company URL (tests intake adaptation)

### Validation Protocol

For each run:
1. Take all Tier 1 candidates (score >= 70)
2. Human reviewer checks each candidate against the benchmark role:
   - **Strong fit:** Would recommend for interview (counts as TRUE POSITIVE)
   - **Weak fit:** Borderline, wouldn't recommend (counts as FALSE POSITIVE)
   - **Bad fit:** Clearly wrong person (counts as FALSE POSITIVE)
3. Calculate: `precision = true_positives / (true_positives + false_positives)`
4. Target: precision >= 0.70 across all 3 runs

### What to Check Per Candidate

- [ ] Does the narrative accurately reflect the evidence?
- [ ] Are the top 3 signals relevant to this role?
- [ ] Is the technical depth score justified by GitHub/code evidence?
- [ ] Is the domain relevance score justified by work history?
- [ ] Are there any fabricated claims in the narrative?

## 2. Dedup Accuracy Testing

### Fixture Data

Use `test-fixtures/dedup-candidates.json` — 11 candidates representing 4 real people + 1 similar-but-different person.

### Automated Test (runs in CI)

The `identity-resolver.test.ts` already covers the core scenarios. Phase 1.7 adds a fixture-based integration test:

```
test('dedup fixture produces expected merge groups')
  Load dedup-candidates.json
  Convert to RawCandidate[]
  Run IdentityResolver.resolve()
  Assert: 5 output candidates (3 merged groups + 1 single + 1 non-merge)
  Assert: Sarah Chen group merged from 3 sources
  Assert: Sara Chen (Google) NOT merged with Sarah Chen (Chainlink)
```

### Live Dedup Test (manual, post-Phase 2)

After adapter-exa is built:
1. Run discovery for the benchmark role
2. Run discovery again with slightly different queries
3. Merge both result sets through IdentityResolver
4. Count duplicates: `duplicate_rate = duplicates_found / total_candidates`
5. Target: duplicate_rate < 0.05

## 3. Cost Per Run Tracking

### Instrumentation (already wired)

Cost tracking is built into the pipeline:
- `SearchPage.costIncurred` — per-page cost from discovery adapters
- `BatchResult.costIncurred` — per-batch cost from enrichment adapters
- `PhaseResult.costIncurred` — per-phase cost reported by handlers
- `CostTracker` — accumulates per-phase and per-adapter costs
- `RunMeta.cost` — final cost snapshot in `run-meta.json`

### What to Measure

After each test run, check `run-meta.json`:
```json
{
  "cost": {
    "totalCost": 3.47,
    "perPhase": { "discover": 2.10, "enrich": 0.87, "score": 0.50 },
    "perAdapter": { "exa": 2.10, "github": 0.00, "ai": 0.50 }
  }
}
```

### Target

- **Typical 50-candidate search:** < $5.00 total (Exa + AI)
- **Budget gate:** Pipeline aborts if `maxCostUsd` is exceeded
- Track cost per candidate: `totalCost / candidateCount` — target < $0.10/candidate

## 5. Role Variations for Extended Testing

After validating with the benchmark role, test with these variations:

1. **Frontend Engineer at a fintech** — different skill signals, same industry
2. **DevRel/Developer Advocate** — community signals > code signals, different weight balance
3. **ML Engineer at a healthcare startup** — niche domain, Python-heavy

These roles test different scoring weight profiles and adapter signal priorities.
