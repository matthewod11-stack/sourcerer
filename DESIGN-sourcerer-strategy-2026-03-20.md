# Design: Sourcerer — Product Strategy & Roadmap Diagnostic

Generated by /office-hours on 2026-03-20
Status: APPROVED
Mode: Startup (open-source + services hybrid)

## Problem Statement

Startup founders and technical hiring managers spend $10K+/yr on LinkedIn Recruiter (bad for niche roles), $30-60K per agency hire (slow, mixed results), and hours of founder time manually searching GitHub/Twitter/Google — all to find the same needle-in-the-haystack candidates. The status quo is expensive AND bad. The harder the hire, the worse the existing tools perform.

## Demand Evidence

- Two paid client implementations already shipped (Lunar Labs, Agoric)
- Both clients were using a combination of LinkedIn Recruiter, agencies, and manual search before
- The builder (you) is the sourcer — you run the tool, deliver scored candidates, clients make hiring decisions
- No waitlist or signups needed — demand is proven by completed work

## Status Quo

Clients cobble together:
- **LinkedIn Recruiter** ($10K+/yr) — works for common roles, fails for niche technical/crypto/AI hires
- **Recruiting agencies** (20-30% of first-year salary) — $30-60K per hire, slow, often miss on cultural fit for startups
- **Manual search** — founders on GitHub/Twitter/Google, tracking in spreadsheets, 10-20 hours per role
- **Network asks** — "know anyone?" posts on Twitter/Slack/Discord

Total cost per hard-to-fill role: $15K-60K+ in direct costs, plus weeks of founder distraction.

## Target User & Narrowest Wedge

**Primary user (V1): You.** The CLI is operational infrastructure for your sourcing service. Every design decision should pass the test: "does this make my next client engagement faster?"

**Secondary user (V1+): Technical startup founder.** Open-source distribution attracts them. Some will self-serve (free). Some will hire you to run it (paid). The CLI serves both paths.

**Narrowest wedge:** The tool at Phase 2 completion (Exa + GitHub + scoring + JSON/markdown output) is already usable for a real engagement. Don't wait for Phase 7.

## Premises

1. The CLI is operational leverage for a services business, not a standalone product. First buyer = you.
2. Done-for-you is the revenue model. The CLI reduces delivery cost per engagement from "build a custom repo" to "run sourcerer init with the client."
3. Portability across clients IS the product requirement. Same tool, different configs — that's why the monorepo/adapter architecture exists.
4. Open-source distribution is marketing. It attracts founders who either self-serve (free awareness) or hire you (paid).
5. The "self-serve technical founder" buyer is a validated bonus, not the V1 target.

## Approaches Considered

### Approach A: Ship the Engine, Sell the Driving (Selected)
- Build the CLI as designed (7-phase roadmap is solid)
- Interleave with real client engagements starting at Phase 2
- Each engagement validates the next phase with real data
- Revenue from day one
- Effort: L | Risk: Low

### Approach B: Full Build, Then Sell
- Execute complete roadmap, then launch
- Effort: XL | Risk: High (months without validation)

### Approach C: Productized Service First
- Package the service now, build CLI later after 10+ engagements
- Effort: S+L | Risk: Medium (feels like going backward)

## Recommended Approach

**Approach A.** The roadmap is validated and well-structured. The key insight is that the CLI and the service are the same thing — portability across clients is the job. Build methodically, but start using it with real clients the moment Phase 2 lands.

## Roadmap Diagnostic

The existing roadmap (`docs/roadmap.md`) is strong after validation. Key observations:

**What's right:**
- Phase sequencing is correct (foundation → adapters → intake → enrichment → scoring → output)
- Parallel agent boundaries are clean and well-defined
- Acceptance criteria are specific and testable (after validation fixes)
- Identity resolution and evidence grounding are treated as first-class design problems

**What to watch:**
- **Phase 2 is your "minimum usable for a client engagement" milestone.** Treat it that way. Run a real (paid or favor) sourcing engagement at Phase 2 completion. The feedback will be worth more than any spec review.
- **Phase 3 (Intake Engine) is the biggest risk.** It's where the "magic" lives but also where scope can explode. The conversation engine, content research subsystem, and company intelligence analysis are each independently complex. Watch for it taking 2x the estimated sessions.
- **Phase 5 (Scoring/Grounding) is the hardest technical problem.** LLM grounding constraints sound clean in the spec but are notoriously hard to enforce in practice. The roadmap correctly adds an early grounding test in Phase 3C — make sure this actually happens.
- **Google Sheets deferral was the right call.** OAuth complexity would have blocked Phase 6 for weeks.

**What's missing from the roadmap (not the tool):**
- **A "client engagement" milestone at Phase 2.** Not a checkbox — an actual commitment to run a sourcing engagement with the tool at Phase 2 completion.
- **A pricing model.** How much do you charge for a done-for-you engagement? Per role? Per month? This doesn't need to be in the roadmap, but it should be figured out before Phase 2 ends.
- **A "show the tool" milestone.** When do you first show Sourcerer to a technical founder who isn't paying you? Open-source launch doesn't need to be Phase 7 — a GitHub repo with a README and "under active development" badge after Phase 3 would attract early interest and feedback.

## Open Questions

1. **Pricing:** What do you charge per done-for-you engagement? (Research: agency fees are 20-30% of first-year salary. A tool-powered service should undercut that significantly while maintaining margin.)
2. **First self-serve user:** When do you want a non-you person to run `sourcerer init` for the first time? Phase 4? Phase 6? This affects how much polish the onboarding wizard needs and when.
3. **Open-source timing:** GitHub repo public at Phase 2 (early signal, raw), Phase 4 (functional), or Phase 7 (polished)?

## Success Criteria

1. Phase 2 completion → run a real sourcing engagement using the tool
2. Phase 3 completion → intake conversation is faster than doing it manually
3. Phase 5 completion → scored output is good enough that a client would trust it without you explaining every candidate
4. Phase 7 completion → a technical founder who has never seen the tool can run `sourcerer init` and produce useful results

## The Assignment

**Before you write a single line of Sourcerer code:** Reach out to 3 startup founders or hiring managers who are actively trying to fill a hard-to-hire role. Offer to run a sourcing engagement for them using your existing repos (LunarSource/ymax-sourcing patterns). Charge something — even $500. The point is to have 3 active engagements in progress BY THE TIME you start Phase 1. This gives you:
- Real roles to test every phase against
- Revenue while you build
- Client feedback on what matters vs. what doesn't
- Case studies for when you open-source
